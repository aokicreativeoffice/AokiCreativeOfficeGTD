# 命名規則（日本語版）

## 1. 基本方針

```text
① 日本語で自然に読める
② 接尾語は最小限
③ 衝突時のみ明示
④ レイヤーは構造で管理
```

## 2. レイヤー別命名ルール
### 2.1 原理（Principle）
#### ルール
〜の原理
#### 例
- フィードバック原理
- 最適化原理
- 因果原理
#### 理由
- 抽象度が高く、明示しないと区別不能
- 学術的にも自然
### 2.2 メカニズム（Mechanism）
### ルール
- 名詞＋動詞的表現（接尾語なし）
#### 例
- 情報拡散
- 資源配分
- 均衡形成
- 誤差修正
- 正のフィードバック
#### 理由
- 動きそのものなので自然文で表現可能
- 「メカニズム」は冗長
#### 例外（衝突時）
- 〜メカニズム
#### 例：
- フィードバックメカニズム（曖昧回避用）
## 2.3 構造（Structure / Rule）
#### ルール
- 名詞（必要なら構造・制約・条件を付与）
#### 例
- ネットワーク
- 階層構造
- フィードバックループ
- 予算制約
- 均衡条件
#### 判断基準
|状況|命名|
|---|---|
|単独で明確|ネットワーク|
|曖昧|ネットワーク構造|
2.4 パターン（Pattern）
#### ルール
- 現象名（接尾語なし）
#### 例
- ロックイン
- バブル
- 崩壊
- カスケード
- フリーライド
#### 禁止
❌ ロックインパターン
❌ バブルパターン
###  2.5 分析（Analysis）
#### ルール
- 〜分析
#### 例
- ネットワーク分析
- 因果分析
- 構造分析
- フィールドワーク分析
### 2.6 フレームワーク（Framework）
#### ルール
- 固有名詞 or 〜フレームワーク
#### 例
- SWOT分析
- 3C
- ファイブフォース
- 価値連鎖
### 2.7 方法（Method）
#### ルール
- 〜法 
- 〜手法
#### 例
- 仮説検証法
- 観察法
- インタビュー法
### 2.8 ツール（Tool）
#### ルール
- 固有名詞
#### 例
- Python
- Notion
- QGIS
## 3. ハブノード（最重要）
### 3.1 命名
- 接尾語なし
#### 例
- フィードバック
- 均衡
- 最適化
- ネットワーク
### 3.2 構造
概念（ハブ）
├ 原理
├ メカニズム
├ 構造
├ パターン
### 3.3 例：フィードバック
フィードバック
├ フィードバック原理
├ フィードバック安定化
├ 正のフィードバック
├ フィードバックループ
├ 自己強化スパイラル
## 4. 衝突回避ルール
### 4.1 発生条件
-同一語が複数レイヤーに存在する場合
### 4.2 解決手順
1. ハブを作る
2. 各レイヤーに分解
3. 必要なものだけ接尾語
4. YAML補助
命名をシンプルにする代わりに：
```
type: mechanism
layer: academic
```
## 5. 最短ルール
- 原理 → 必ず「〜原理」
- メカニズム → そのまま
- 構造 → 必要なら「〜構造」
- パターン → そのまま
- ハブ → そのまま
## 6. NG例
### NG1
- フィードバック（全部入れる）
### NG2
- フィードバック原理構造メカニズム
### NG3
- ロックインパターン
## 7. 最終定義
- 名前は自然に
- 意味は構造で管理する
